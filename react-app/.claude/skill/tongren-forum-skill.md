# 同人论坛网站 Skill

## 使用场景

当用户要求开发、修改或设计同人论坛网站相关功能时使用本 skill。项目需求包括登录注册验证码、文字/图片博客发布、Word/PDF/Markdown 文档文字解析、合集管理、点赞评论收藏、搜索功能。

## 项目目标

实现一个面向同人创作者和读者的论坛网站，核心能力如下：

1. 用户可以注册、登录，并通过验证码降低机器人注册和暴力登录风险。
2. 用户可以发布文字和图片博客，也可以上传 `.doc`、`.docx`、`.pdf`、`.md` 文档并解析其中的文字内容作为博客正文。
3. 用户可以创建合集，将多篇博客归档管理。
4. 用户可以对内容点赞、评论、收藏。
5. 用户可以通过关键词搜索文章、合集、作者和标签。

## 技术约束

- 如果修改 Expo 或 React Native 相关代码，先查看当前项目依赖版本，并参考 Expo v54 文档：https://docs.expo.dev/versions/v54.0.0/。
- 优先保持实现简单直接，不引入不必要的框架或复杂抽象。
- 前端交互需要兼顾桌面端和移动端可用性。
- 涉及上传、鉴权、验证码、文件解析时，优先考虑安全边界和服务端校验。
- 不要把验证码、上传限制、文件类型校验只放在前端。

## 功能拆分

### 1. 登录注册验证码

- 注册表单字段：用户名、邮箱、密码、确认密码、验证码。
- 登录表单字段：邮箱或用户名、密码、验证码。
- 验证码可以先使用图片验证码或邮箱验证码，后续可替换为第三方验证码服务。
- 服务端需要校验验证码有效期、使用次数和错误次数。
- 密码必须加密存储，禁止明文保存。

### 2. 发布文字、图片和解析文档文字

- 发布内容支持标题、正文、封面图、标签、可见性。
- 正文支持 Markdown 编辑和预览。
- 图片上传需要限制大小、类型和数量。
- 文档上传的实际用途是解析文字内容，并把解析出的文本填入或追加到博客正文。
- 文档解析支持 Word、PDF、Markdown：`.doc`、`.docx`、`.pdf`、`.md`。
- 文档解析应在后端执行，前端只负责上传文件、展示解析结果和错误提示。
- 上传后可按需要保存原文件和文件元信息：原始文件名、MIME 类型、大小、存储地址、上传用户、创建时间。

### 3. 合集管理文档

- 用户可以创建、编辑、删除合集。
- 合集字段建议包含：标题、简介、封面、标签、排序方式、是否公开。
- 合集可以关联多篇博客。
- 支持调整合集内内容顺序。
- 删除合集时默认不删除原始博客或文档，除非用户明确选择。

### 4. 点赞、评论、收藏

- 点赞和收藏应防止同一用户重复创建同一关系。
- 评论支持文章评论和回复评论。
- 评论需要保存作者、目标内容、父评论、正文、创建时间、状态。
- 内容列表和详情页应展示点赞数、评论数、收藏状态。
- 删除内容时需要处理关联的点赞、评论、收藏数据。

### 5. 搜索功能

- 搜索范围包括标题、正文摘要、标签、作者、合集名称。
- 搜索结果需要区分内容类型，例如文章、合集、用户。
- 支持按时间、热度、相关性排序。
- 初期可以使用数据库模糊查询；数据量增大后再接入全文搜索服务。

## 后端设计

后端优先采用简单、低维护成本的方案：`FastAPI + SQLite + SQLAlchemy + JWT + 本地文件上传`。

### 技术选型

- 后端框架：`FastAPI`。
- 数据库：`SQLite`，适合本地开发、原型验证和小型部署。
- ORM：`SQLAlchemy`。
- 数据迁移：第一版可用 `Base.metadata.create_all(bind=engine)` 自动建表；结构稳定后再接入 `Alembic`。
- 鉴权：`JWT` 登录态。
- 密码加密：`passlib[bcrypt]`。
- 文件存储：本地 `backend/uploads/` 目录。
- API 文档：FastAPI 自带 `/docs`。

### 后端目录建议

```text
backend/
  app/
    main.py
    database.py
    models.py
    schemas.py
    auth.py
    routers/
      auth.py
      posts.py
      uploads.py
      collections.py
      comments.py
      search.py
  uploads/
  requirements.txt
```

### 接口建议

- `POST /auth/register`: 注册，校验验证码并创建用户。
- `POST /auth/login`: 登录，校验验证码并返回 JWT。
- `GET /auth/captcha`: 获取验证码。
- `POST /posts`: 创建博客。
- `GET /posts`: 获取博客列表。
- `GET /posts/{post_id}`: 获取博客详情。
- `POST /uploads/images`: 上传图片。
- `POST /uploads/parse-document`: 上传 Word、PDF 或 Markdown，并返回解析出的文本。
- `POST /collections`: 创建合集。
- `POST /collections/{collection_id}/items`: 添加博客到合集。
- `POST /posts/{post_id}/like`: 点赞或取消点赞。
- `POST /posts/{post_id}/favorite`: 收藏或取消收藏。
- `POST /posts/{post_id}/comments`: 创建评论。
- `GET /search`: 搜索博客、合集、作者和标签。

### 文档文字解析建议

- `.md`: 直接按文本读取，注意编码处理。
- `.docx`: 可使用 `python-docx` 提取段落文字。
- `.pdf`: 可使用 `pypdf` 或 `pdfplumber` 提取文字。
- `.doc`: 优先提示转换为 `.docx`；如必须支持，可后续接入 `antiword`、LibreOffice 转换或其他解析服务。
- 解析失败时返回明确错误，不要静默创建空正文。

## 建议数据模型

可根据项目实际技术栈调整命名。

- `User`: 用户账号、邮箱、密码哈希、头像、简介、状态。
- `Captcha`: 验证码、用途、过期时间、是否已使用、失败次数。
- `Post`: 标题、正文、作者、封面、标签、状态、可见性、统计字段。
- `Asset`: 上传图片或原始文档元信息，关联用户和内容。
- `Collection`: 合集标题、简介、封面、作者、可见性。
- `CollectionItem`: 合集和内容的关联关系及排序。
- `Like`: 用户和内容的点赞关系。
- `Favorite`: 用户和内容的收藏关系。
- `Comment`: 评论正文、作者、目标内容、父评论、状态。

## 页面建议

- 主页面：首页入口，顶部放搜索框，搜索框下方包含 `发现` 和 `关注` 两个内容区。
- 底栏导航：包含 `首页`、`加号`、`我的` 三个入口。
- 加号入口：从主页面底栏点击 `加号` 跳转到上传页面，用于发布博客。
- 上传页面：标题、正文编辑器、图片上传、文档文字解析、标签选择。
- 我的主页：从主页面底栏点击 `我的` 跳转，展示用户资料、发布内容、收藏、合集。
- 登录页：账号密码登录和验证码。
- 注册页：账号注册和验证码。
- 文章详情页：正文、图片、点赞、评论、收藏。
- 合集页：合集信息、博客列表、排序管理。
- 搜索页：搜索框、筛选条件、结果列表。

## 初步页面设计

- 主页面：顶部搜索框；主体包含 `发现` 和 `关注` 两个区；底栏包含 `首页`、`加号`、`我的` 三个区。
- 上传页面：从主页面点击底栏 `加号` 按钮跳转，用于上传或发布博客。
- 我的主页：从主页面点击底栏 `我的` 跳转，用于查看和管理个人内容。

## 开发流程

1. 先检查当前项目结构、路由方式、状态管理、UI 组件和后端接口约定。
2. 再确定本次需求属于页面、组件、接口、数据模型还是样式调整。
3. 优先实现最小可用版本，保证核心流程能跑通。
4. 对登录、注册、上传、评论等功能补充必要的错误处理和加载状态。
5. 完成后运行项目已有的类型检查、lint、测试或构建命令。

## 验收标准

- 注册和登录流程能正确校验验证码。
- 用户能成功发布文字内容、图片内容，并能从支持格式的文档中解析文字。
- 用户能创建合集，并把内容加入合集。
- 点赞、评论、收藏状态能正确展示和切换。
- 搜索能返回相关内容，并在无结果时展示友好提示。
- 非法文件类型、超大文件、未登录操作、重复点赞收藏等情况有明确处理。
